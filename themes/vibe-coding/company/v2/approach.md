# Approach

Cabinet is what Relay has to become if the 26-week playbook works.

Relay guides the founder through the journey. Cabinet makes that journey persistent. It keeps the company running before the generally available product launch, and it keeps running after launch when the founder no longer wants to personally hold every thread in their head.

This is forward-looking. It was not obvious inside the CFO-suite experiment. The experiment taught me something else: I have a tremendous amount of patience for researching, improving, and building a product, but I cannot realistically run a second company by hand forever. If Thin Wedge works, I need a way to migrate it into a company runtime.

That is Cabinet.

Cabinet is Relay as a service, and as a vibe-holdable project. It stores durable company state in a database plus a Git repo. The database holds live operating state. The Git repo gives the company a versioned memory of plans, decisions, artifacts, policies, evals, and roadmap changes.

The founder is still the CEO. They own the final authoritative decisions. They can fractionalize responsibility and delegate specific authority to agents, but Cabinet should not pretend that the company has no owner. It is a cabinet of ministers, not a replacement founder.

The most important thing Cabinet does is trigger the right agents and track the roadmap. This is not a normal static roadmap. It starts lightly branched, then deepens or broadens as the company learns more. Cabinet should know which branch of the company journey is active, which agents own which work, and which decisions need the founder.
